# Архитектура системы проверки неймингов

**Сервис проверки неймингов**  
*Индустриальный проект МТС*

---

## Общая идея

Система предназначена для автоматической проверки нейминга и снижения нагрузки на юридический отдел. В актуальной постановке у нее есть два основных сценария:
- проверка нового текстового нейминга на возможность регистрации в РФ;
- автоматическая проверка неправомерного использования товарного знака.

Дополнительно рассматривается расширение системы на проверку логотипов, возможно в виде отдельного сервиса.

## Основные компоненты

### 1. REST API
**Роль**: Точка входа в систему  
**Функции**:
- Принимает запросы от юристов
- Валидирует входные данные (нейминг + класс МКТУ)
- Возвращает результаты проверки
- Поддерживает как минимум два сценария:
  - проверка нового нейминга;
  - проверка возможного нарушения

### 2. Naming Check Service
**Роль**: Основная бизнес-логика системы  
**Функции**:
- Координирует процесс проверки нейминга
- Выполняет поиск похожих названий
- Сравнивает и ранжирует результаты
- Вычисляет метрики сходства
- Работает в нескольких режимах:
  - `registration check` — поиск top-200 конфликтующих обозначений;
  - `infringement check (text)` — сравнение одного нейминга с другим;
  - в расширении: `infringement check (logo)` — сравнение логотипов.

### 3. Data Storage (Хранилище данных)

#### 3.1. Naming Database
**Роль**: Основная база товарных знаков и связанных сущностей  
**Содержит**:
- Названия товарных знаков
- Классы МКТУ
- Даты регистрации
- Метаданные
- Дополнительные поля для предобработки, например `mark_disclaimer`

#### 3.2. Embedding Model
**Роль**: Обученная модель для генерации векторных представлений  
**Функции**:
- Преобразует названия в векторные представления
- Отражает семантические свойства названий
- Поддерживает русский и английский языки
- Разворачивается локально, без доступа во внешний интернет

#### 3.3. Embedding Dataset
**Роль**: Предвычисленные эмбеддинги для быстрого поиска  
**Содержит**:
- Векторные представления всех названий из базы данных
- Используется для быстрого семантического поиска без повторной обработки

#### 3.4. Search Indexes
**Роль**: Индексы для быстрого поиска  
**Типы индексов**:
- **Фонетический индекс**: Поиск по звучанию (адаптированные `Soundex/Metaphone`)
- **Лингвистический индекс**: Поиск по написанию (`n-grams`, префиксы, строковые метрики)
- **Векторный индекс**: Поиск ближайших эмбеддингов

#### 3.5. Preprocessed Naming Store
**Роль**: Слой предобработанных текстовых обозначений  
**Содержит**:
- очищенные и нормализованные названия;
- извлеченный основной словесный элемент;
- удаленные неохраняемые элементы;
- подготовленные формы для быстрого поиска и сравнения.

## Пошаговый процесс работы

### Шаг 1: Запрос от юриста
Юрист отправляет запрос через REST API:
- Название для проверки (например, "PROBIMAX")
- Класс МКТУ (в основном сценарии используется для фильтрации результатов)
- Тип сценария:
  - регистрация;
  - проверка нарушения;
  - в будущем, возможно, логотип.

### Шаг 2: API принимает запрос
- REST API валидирует входные данные
- Передает запрос в сервис проверки

### Шаг 3: Сервис проверки обрабатывает запрос

Сервис выполняет несколько операций для поиска похожих названий:

#### 3.0. Предфильтрация по МКТУ
- Для сценария регистрации сначала выбираются наименования, чьи классы МКТУ совпадают с заданными.
- Это сокращает число кандидатов и делает проверку ближе к юридической практике.

#### 3.1. Генерация эмбеддинга (семантический поиск)
- Использует **Embedding Model** для преобразования проверяемого названия в вектор
- Вектор отражает семантические свойства названия
- Ищет похожие векторы в **Embedding Dataset** (предвычисленные эмбеддинги)
- Находит названия с близким смыслом

#### 3.2. Фонетический и лингвистический поиск
- Использует **Search Indexes** для быстрого поиска:
  - **Фонетический индекс**: Находит названия, похожие по звучанию (например, "EUROPLEX" и "EUROFLEX")
  - **Лингвистический индекс**: Находит названия, похожие по написанию (расстояние Левенштейна, n-grams)

#### 3.3. Получение данных о товарных знаках
- Из **Naming Database** получает полную информацию о найденных товарных знаках:
  - Название
  - Класс МКТУ
  - Дата регистрации
  - Другие метаданные

#### 3.4. Использование предобработанных данных
- Перед сравнением используется очищенная форма обозначения из **Preprocessed Naming Store**.
- Из длинных описательных полей выделяется основной нейминг.
- Удаляются:
  - цифры и числа без различительной способности;
  - общепринятые аббревиатуры;
  - родовые/описательные обозначения;
  - элементы из `mark_disclaimer`.

### Шаг 4: Сравнение и ранжирование
- Сервис сравнивает проверяемое название с найденными кандидатами
- Вычисляет метрики сходства:
  - **Фонетическое сходство**: По звучанию
  - **Графическое сходство**: По написанию
  - **Семантическое сходство**: По смыслу
  - **Лингвистическое сходство**: По строковым метрикам
- Формирует **итоговый процент сходства** по формуле, подобранной экспериментально на основе тестирования сервиса
- Учитывает юридические правила:
  - Игнорирует неохраняемые элементы (цифры, аббревиатуры ООО, родовые термины)
  - Фокусируется на охраняемых (доминирующих) элементах
- Учитывает, что однокоренные слова не всегда должны считаться семантически сходными
- Ранжирует результаты по релевантности

### Шаг 5: Возврат результатов
- В сценарии регистрации сервис возвращает в API список из `~200` наиболее конфликтующих названий
- В сценарии нарушения текстового знака сервис может возвращать процент сходства для пары обозначений
- API отправляет результаты юристу
- Юрист получает список потенциальных конфликтов с метриками сходства

### Шаг 5.1: Асинхронный внешний этап
- Внешний поиск выполняется как отдельный Stage 2 после внутреннего ответа.
- Stage 2 доставляется только через `webhook`.
- Допустим частичный внешний результат, если часть источников не ответила вовремя.
- Порядок доставки внешних обновлений не важен.
- Для внешних источников используются до `3` попыток с `exponential backoff` и timeout `60s` на один источник.
- Дедупликация внешних результатов выполняется по ключу `naming + MKTU`.

## Ключевые особенности архитектуры

### 0. Офлайн-развертывание
- Сервис должен работать во внутренней сети МТС.
- Во время работы сервиса запрещены обращения во внешний интернет.
- Модели, словари и зависимости должны быть заранее скачаны и включены в проект.

### 1. Предвычисленные эмбеддинги
- Эмбеддинги для всех названий в базе предвычисляются заранее
- Хранятся в отдельном датасете
- При проверке нового названия его эмбеддинг вычисляется один раз
- Поиск выполняется по предвычисленным эмбеддингам

**Преимущество**: Быстрый семантический поиск без необходимости обрабатывать всю базу через модель при каждом запросе.

### 2. Множественные индексы
- Фонетический индекс для поиска по звучанию
- Лингвистический индекс для поиска по написанию
- Векторный индекс для семантического поиска

**Преимущество**: Быстрый поиск по разным типам сходства.

### 3. Параллельный поиск
- Семантический поиск (по эмбеддингам)
- Фонетический поиск (по индексам)
- Лингвистический поиск (по индексам)

Выполняются параллельно для оптимизации производительности.

### 4. Агрегация источников
- Результаты могут приходить из нескольких каналов:
  - внутренняя база товарных знаков;
  - маркетплейсы приложений;
  - внешние источники и реестры, предварительно собранные во внутренний контур.
- После этого они объединяются в единый список наиболее релевантных совпадений.
- Во внешнем async-контуре допускаются частичные и unordered результаты с последующей дедупликацией и merge.

### 5. Комбинированная оценка
- Результаты из разных источников объединяются
- Вычисляется интегральная метрика сходства
- Учитываются юридические правила и критерии

### 6. Возможность разделения на два сервиса
- Если требования по времени отклика окажутся слишком жесткими, архитектура допускает разделение на:
  - сервис сравнения текстовых обозначений;
  - сервис сравнения логотипов.

## Результат работы системы

Юрист получает:
- **Список из ~200 названий**, которые могут помешать регистрации
- **Сортировка по степени конфликта** (от наиболее похожих к менее похожим)
- **Процент сходства** и, при необходимости, детализацию по типам:
  - Фонетическое сходство
  - Семантическое сходство
  - Графическое сходство
  - Лингвистическое сходство
  - Интегральная метрика (общее впечатление)

Дополнительно система может использоваться для:
- сравнения одного текстового обозначения с другим;
- сравнения логотипов (в отдельном контуре).

## Связь с Ubiquitous Language

Архитектура использует термины из [Ubiquitous Language](ubiquitous_language.md):
- **Нейминг (Naming)** — проверяемое название
- **Товарный знак (Trademark)** — элементы из базы данных
- **Сходство до степени смешения (Confusing Similarity)** — основной критерий для блокировки регистрации
- **Эмбеддинг (Embedding)** — векторное представление для семантического поиска
- **Метрики сходства** — фонетическая, графическая, семантическая, лингвистическая
- **Конфликтующий нейминг (Conflicting Naming)** — результат работы системы
